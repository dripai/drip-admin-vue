# 后台管理系统前端设计提示词模板

下面提示词用于让 AI 生成一套完整、可落地、精简实用的后台管理系统前端方案或前端项目代码。适用于企业内部后台、运营后台、权限系统、配置平台、数据管理平台、业务中台等管理端产品。

```text
你是一名资深前端架构师、Vue 工程专家和企业级后台管理系统产品设计专家。请基于当前主流前端工程实践，为我设计并实现一套“后台管理通用框架前端系统”。

一、总体目标

设计一套精简、实用、可复用、可快速迭代的后台管理系统前端框架，用于承载企业内部系统、数据管理平台、运营后台、权限系统、业务配置平台等场景。

系统设计要求：
1. 功能描述必须完整、具体、可落地，不要大而空洞的页面描述。
2. 页面必须围绕真实后台管理业务场景设计。
3. 不设计无业务价值的展示页、大图、营销式首屏、无用提示信息、无用统计卡片。
4. 不保留过时兼容参数、备用请求格式、隐式 fallback 请求路径。
5. 重复能力必须封装为通用组件、组合式函数、请求工具、权限工具或配置化模块。
6. 登录、布局、动态路由、菜单、按钮权限、用户管理、角色管理、菜单权限管理等基础能力必须完整。
7. 每个页面需要明确页面职责、字段、筛选项、表格列、表单项、操作按钮、权限点、校验规则、异常状态。
8. 页面结构清晰，不把所有业务都堆进一个超大组件。
9. 缺少必要接口、配置或数据时，显示明确错误或空状态，不要静默降级或伪造数据。
10. 设计结果应能直接指导前端开发、接口联调和功能验收。

二、技术栈约束

在满足兼容性的前提下，尽可能使用当前稳定的新版本框架。

默认技术栈：
- Vue 3
- TypeScript
- Vite
- Ant Design Vue
- Vue Router
- Pinia
- Axios
- ESLint
- Prettier
- Vitest
- Playwright 可选，用于端到端测试

要求：
1. 使用 Composition API 和 `<script setup lang="ts">`。
2. 所有接口请求、响应、表单模型、表格数据必须定义 TypeScript 类型。
3. UI 组件基于 Ant Design Vue，不要混用多套 UI 框架。
4. 请求层统一使用 Axios 封装，不再提供 fetch 兼容分支。
5. 图标统一基于 `@ant-design/icons-vue` 管理，并封装菜单图标选择能力。
6. 样式使用 Sass + scoped style + 全局样式变量，优先复用 Ant Design Vue 设计变量和组件能力。
7. 样式保持企业后台风格：清晰、克制、高密度、可扫描。
8. 不要做营销站、落地页、hero 大图、装饰性插画、无意义动效。
9. 不要为了“高级感”引入复杂动画和无业务价值的视觉元素。

三、产品范围

第一版后台前端必须形成完整闭环：
登录 -> 获取当前用户信息 -> 获取菜单权限 -> 渲染布局和动态路由 -> 用户管理 -> 角色授权 -> 权限生效 -> 操作日志可查。

必须包含以下模块：

1. 认证与会话
   - 登录页。
   - 退出登录。
   - token 存储。
   - token 过期处理。
   - 获取当前用户信息。
   - 获取当前用户菜单和权限码。
   - 修改当前用户密码。

2. 主布局
   - 左侧菜单。
   - 顶部栏。
   - 面包屑。
   - 用户下拉菜单。
   - 页面内容区。
   - 菜单折叠。
   - 动态路由加载。

权限模型要求：
- 使用 RBAC 权限模型，以用户、角色、菜单、按钮权限码为核心。
- 菜单类型分为目录、页面、按钮。页面类型在后端可映射为 MENU，按钮类型只作为权限点，不进入左侧菜单。
- 权限控制必须到按钮级别，页面入口和按钮操作都要有明确权限码。
- 前端只做路由和按钮显示控制，真实权限校验由后端接口完成。

3. 用户管理
   - 查询筛选。
   - 分页表格。
   - 新增用户。
   - 编辑用户。
   - 删除用户。

4. 角色管理

5. 菜单与权限管理

   - 新增菜单。
   - 新增按钮权限。
   - 编辑菜单。
   - 删除菜单。
   - 调整排序。
   - 启用 / 禁用。

6. 部门管理
   - 部门树。
   - 部门详情。
   - 新增部门。
   - 编辑部门。
   - 删除部门。
   - 启用 / 禁用。
   - 调整上级部门。

7. 字典管理
   - 字典类型列表。
   - 字典项列表。
   - 新增、编辑、删除字典类型。
   - 新增、编辑、删除字典项。
   - 启用 / 禁用字典项。
   - 刷新字典缓存。

8. 系统配置
   - 配置列表。

   - 编辑配置。

9. 日志管理
   - 登录日志分页列表。
   - 登录日志详情。
   - 业务操作日志分页列表。
   - 业务操作日志详情。
   - 按操作人、模块、类型、状态、时间范围查询。
   - 敏感字段脱敏展示。
   - 只展示后端已落表的关键操作日志。
   - 登录、登出和业务新增、编辑、删除、启用、禁用、授权、重置密码等更新类操作才进入日志表。
   - 普通查询、列表刷新、详情查看等高频只读访问不进入日志表，可由后端写入文件日志，避免数据库日志过度膨胀。
   - 业务操作日志承担基础审计查询职责，不单独设计重复的审计报表模块。

四、MVP 优先级

请按以下优先级设计和实现，只保留 P0 和 P1，避免第一版过度膨胀。

P0：第一版必须完成
- 登录页。
- 主布局。

- 当前用户信息。
- 用户管理。

- 菜单权限管理。
- 登录日志列表。
- 业务操作日志列表。
- 统一请求、统一错误处理、统一空状态、统一 loading。

P1：常用增强
- 部门管理。
- 字典管理。
- 系统配置。
- 修改密码。
- 数据权限配置。
- 查询条件持久化。
- 表格列显示控制。

- 批量导入 / 导出通用组件预留，不设计具体业务页面。

不单独设置 P2。后续新业务必须先明确业务场景，再进入新的迭代清单。

审计报表处理原则：
- 不单独设计审计报表模块，避免和业务操作日志重复。
- 登录日志用于追踪登录、登出、登录失败、登录 IP、登录设备等会话事件。
- 业务操作日志用于追踪新增、编辑、删除、启用、禁用、授权、重置密码、备份、恢复等关键变更事件。
- 业务操作日志可作为基础审计查询页面，通过筛选、详情和导出能力满足常规审计需要。

每个优先级必须输出：
- 页面范围。
- 组件范围。
- 接口依赖。
- 权限点。
- 验收标准。

五、页面设计原则

后台页面必须精简、实用、信息密度适中。

必须遵守：
1. 页面首屏直接展示核心操作区，不做欢迎页、介绍页、宣传图。
2. 列表页结构统一：查询表单 -> 操作按钮 -> 数据表格 -> 分页。
3. 新增和编辑优先使用抽屉或弹窗，复杂表单可以使用独立页面。
4. 删除、禁用、重置密码、授权等高风险操作必须二次确认。
5. 空数据使用简短空状态，不写冗长说明。
6. 错误提示必须明确可行动，不展示后端堆栈。
7. 表格列只展示管理决策需要的信息，不塞入低价值字段。
8. 操作列按钮必须按权限显示。
9. 表单字段必须有明确校验，不依赖后端兜底。
10. 不使用无意义图表、无业务价值统计卡片、大面积背景图。

六、布局设计

请设计统一后台布局：

1. 登录页
   - 居中登录表单。
   - 用户名。
   - 密码。
   - 登录按钮。
   - loading 状态。
   - 登录失败提示。
   - 不展示无用宣传语和大图。

2. 主框架
   - 左侧菜单宽度固定，可折叠。
   - 顶部栏显示系统名称、折叠按钮、用户信息和退出登录。
   - 内容区包含面包屑和页面主体。
   - 页面主体使用统一内边距。
   - 列表页操作区保持一致。

3. 动态菜单
   - 根据后端菜单树生成路由。
   - DIRECTORY 作为菜单分组。
   - MENU 作为可点击页面。
   - BUTTON 不进入左侧菜单，只作为权限码。
   - 菜单按 sort 升序显示。
   - hidden / visible 状态必须生效。

八、接口联调契约

必须基于后端接口设计前端请求层。

统一响应结构：
{
  "code": 0,
  "message": "success",
  "data": {}
}

分页响应：
{
  "list": [],
  "total": 0,
  "page": 1,
  "pageSize": 20
}

必须封装：
1. request 实例。
2. 请求 baseURL。
3. token 注入。
4. 401 自动跳转登录。
5. 403 显示无权限提示。
6. 业务错误统一 message 提示。
7. 文件下载可选。
8. 请求取消可选。

Axios 封装要求：
- 只使用 Axios 作为请求库。
- 统一创建一个 request 实例。
- 统一设置 baseURL、timeout、headers。
- 统一处理 token 注入、业务错误、HTTP 错误、401、403。
- API 模块只能调用统一 request 实例，不允许页面直接调用 axios。

不要设计：
- 多套请求库。
- 多种响应结构兼容。
- 静默 fallback 到 mock 数据。
- 前端自行拼接数据权限条件。

九、权限设计

前端权限必须依赖后端返回的数据，并遵循 RBAC 权限模型。

RBAC 规则：
1. 用户通过角色获得权限。
2. 角色关联菜单和按钮权限。
3. 菜单类型分为 DIRECTORY、PAGE、BUTTON，其中 PAGE 可映射后端 MENU 类型。
4. DIRECTORY 只作为菜单分组，不对应具体页面。
5. PAGE 对应可访问页面和路由。
6. BUTTON 对应页面内操作权限，不进入左侧菜单。
7. 权限必须控制到按钮级别，按钮权限码格式建议为 `module:resource:action`。
8. 前端按钮显示基于 permissionCode 判断，后端接口仍必须校验同一权限码。

必须实现：
1. 登录后保存 token。
2. 拉取当前用户信息。
3. 拉取菜单树和权限码。
4. 根据菜单树生成动态路由。
5. 根据权限码控制按钮显示。
6. 路由守卫校验登录状态和路由权限。
7. 退出登录清理 token、用户信息、菜单、权限码。
8. 菜单图标根据后端返回的 icon key 通过 IconRenderer 统一渲染。

规则：
- 前端权限只负责显示控制和路由拦截。
- 后端必须负责真实接口权限。
- 前端不能决定数据权限范围。
- 未匹配路由显示 404。
- 已登录用户访问登录页应跳转首页。
- token 失效应跳转登录页。

3. permissionStore

- 不引入 Tailwind、Bootstrap 或其他 UI / CSS 框架，除非产品明确要求。

必须遵守：
1. 使用 Ant Design Vue 的 Form、Table、Tree、Modal、Drawer、Tag、Switch、Dropdown、Menu、Breadcrumb、Layout 等组件。

3. 操作按钮顺序统一：新增、批量操作、刷新；行内为编辑、授权、禁用、删除。
4. 危险操作使用 danger 风格和二次确认。
5. loading 状态覆盖请求中的表格、表单提交和登录按钮。
6. 表单提交成功后关闭弹窗并刷新列表。
7. 表单提交失败保留用户输入。
8. 查询重置后回到第一页。
9. 分页切换保持当前筛选条件。
10. 长文本字段使用省略和 tooltip。
11. 移动端不是主要目标，但常见笔记本分辨率必须可用。

不要设计：
- 大面积装饰图。
- 首页营销 banner。
- 与业务无关的图表。
- 复杂主题切换。
- 花哨动画。
```
